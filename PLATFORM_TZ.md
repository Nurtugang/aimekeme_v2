# Платформа видеонаблюдения — ТЗ для команды (MVP)

## Контекст

Есть готовый **AI-API** (отдельный репозиторий) с тёплыми моделями. Он принимает
кадры и возвращает результат. Вы строите **платформу** (отдельный репозиторий),
которая подключает камеры, гоняет через AI-API и показывает результат оператору.

**AI-API трогать нельзя** — это зафиксированный контракт (см. его README):
- `POST /detect/fight` — тело `{ "frames": [16 × base64_jpg] }` → `{ label, confidence, processing_ms }`
- `POST /detect/face`  — тело `{ "frame": base64_jpg }` →
  `{ faces:[{box,det_confidence,identity,identity_id,similarity}], count, processing_ms }`
  (`identity` = имя или `"unknown"`; `identity_id` = стабильный ID человека или `null`).
- `GET  /health`

**База лиц живёт в AI-API** (эталоны + эмбеддинги model-specific). Платформа
управляет ею через эти эндпоинты (например, из админки):
- `POST   /faces`             — `multipart/form-data`: `name` + `images` (1..N файлов,
  на каждом одно чёткое лицо) → `{ id, name, created_at, photos }`. Несколько фото
  на человека сильно повышают точность — заливать разные ракурсы/условия.
- `POST   /faces/{id}/images` — догрузить ещё фото человеку → та же запись.
- `GET    /faces`             — список `[{ id, name, created_at, photos }]`.
- `GET    /faces/{id}/image`  — первичное фото человека (image/jpeg).
- `DELETE /faces/{id}`        — удалить человека.

Распознавание (`/detect/face`): модель — ArcFace (`buffalo_l`), порог косинуса 0.42.
Журнал событий платформы пусть ссылается на `identity_id` (стабильный, переживает
переименование), а не на имя.

## Архитектура платформы (MVP, без Redis)

```
[Камера RTSP] ──> [Broker (Python)] ──POST /detect/<module>──> [AI-API]
                       │  (декод ОДИН раз на камеру)
                       ▼
                 [Backend (FastAPI): камеры + события (SQLite) + WebSocket]
                       │
                       ▼
                 [Frontend (React): страницы ниже]
```

## Минимальный объём (features)

1. **Главная страница** — обзор/вход в систему.
2. **Добавление камеры** — форма: имя, RTSP-URL, локация.
3. **Дашборд** — сетка камер; можно выбрать из списка, какие камеры мониторить сейчас.
4. **Меню камеры** — на окне камеры «троеточие» → выбрать модуль(и): `fight`, `face`, ...
5. **Показ результата** — рядом в логах и/или оверлеем на камере (напр. «FIGHT / NORMAL»).
6. **Broker** — связывает RTSP с AI-API (без Redis, in-process очередь / прямой HTTP).

## UI/UX — дизайн-гайдлайны

- **Навигация**: сайдбар слева (фиксированный), не топбар. Сверху вниз:
  Главная → Камеры (с кнопкой "+" и списком активных ниже) → Управление
  камерами → Лэйаут.
- **Плотность**: тесная вёрстка, минимум воздуха. Не делатьcбольшие отступы — это рабочий инструмент оператора, не маркетинг-сайт.
- **Тема**: светлая. Фон белый/светло-серый, текст тёмный.
- **Акцент**: один цвет — циан/тил, как на логотипе ФИТ (примерно
  #2BA9A0). Используется для активных элементов, кнопок, рамки выбранной камеры. Больше цветов не
  вводим.
- **Без градиентов.** Везде плоский цвет.
- **Тексты**: не придумывать копирайт/маркетинговые тексты. Либо
  плейсхолдер (`Текст здесь`), либо нейтральный технический текст
  ("Список камер пуст", "Добавить камеру"). Финальные тексты — отдельная
  итерация, не тратим время сейчас.

### Добавление камеры — превью

После заполнения полей (имя, RTSP-URL, локация) — кнопка **"Проверить
подключение"**. По нажатию: backend пробует открыть RTSP
(`cv2.VideoCapture`), берёт один кадр, отдаёт на фронт как картинку.
Если не открылось за N секунд (таймаут ~5 сек) — показать ошибку
("Не удалось подключиться"), сохранение формы не блокируем — пользователь
может сохранить и разобраться позже.

### Дашборд (страница мониторинга)

**Пустое состояние.** При входе, если активных камер нет — по центру
экрана плейсхолдер с текстом и кнопкой "Добавить камеру".

**Добавление в мониторинг.** По нажатию — список доступных камер
(модальное окно либо инлайн-панель, на выбор разработчика). Выбор камеры
из списка добавляет её в текущий мониторинг.

**Сетка (grid mode).** Поведение зависит от количества активных камер:
- 1 камера → занимает весь экран.
- 2 камеры → экран делится на 2 равные половины.
- 3 камеры → 1 крупная + 2 поменьше (либо равный ряд из 3х — решает
  фронтенд, главное не оставлять пустых ячеек).
- 4 камеры → ровная сетка 2×2.
- 5+ камер → автоматическая сетка (как у большинства NVR-клиентов),
  не усложняем отдельной логикой под каждое число.

**Режим фокуса (focus mode).** Альтернативный режим: одна выбранная
камера крупная, остальные — мелкие миниатюры по краю/снизу. Переключение
между grid/focus — через пункт "Лэйаут" в сайдбаре.

**Управление модулями на камере.** В правом верхнем углу окна камеры —
иконка "три точки". По клику — список модулей (`fight`, `face`, ...) с
переключателями вкл/выкл **для этой конкретной камеры**.

## Сущности БД (SQLite, MVP)

```
User { id, username, password_hash, created_at }

Camera {
  id, user_id (FK -> User),     # камера принадлежит пользователю
  name, rtsp_url, location, enabled,
  modules_enabled (JSON),       # {"fight": true, "face": false, ...}
  created_at
}

UserSettings {

  user_id (FK -> User, unique),
  layout_mode ("grid" | "focus"),
  active_camera_ids (JSON),     # какие камеры сейчас в мониторинге
  focused_camera_id (FK -> Camera, nullable)
}

Event {
  id, camera_id (FK -> Camera), module, label,
  confidence, ts, clip_path?
}
```
Примечания:
- `modules_enabled` — JSON-поле прямо на `Camera`, а не отдельная таблица.
  Проще для MVP; если позже понадобятся параметры конкретного модуля
  (свой threshold на камеру) — выносим в отдельную таблицу тогда.
- `UserSettings` — связь 1:1 с `User`, отдельная таблица вместо полей
  на самом `User`, чтобы не раздувать основную таблицу пользователя.
- Доступ: `Camera.user_id` обеспечивает изоляцию — пользователь видит
  только свои камеры на уровне запроса (`WHERE user_id = current_user.id`).

## Broker — как устроен (без Redis)

- На каждую камеру: `cv2.VideoCapture(rtsp_url)` — **один** декод.
- Для `fight`: скользящее окно `deque(maxlen=16)`, шаг 8 кадров → 16 кадров в base64 → `POST /detect/fight`.
  (Готовая логика окна уже есть в AI-репо: `test/inference.py`, функция `run`.)
- Для `face`/`counting`/`fire`: брать каждый N-й кадр → 1 кадр в base64 → `POST /detect/<module>`.
- Результат → backend → WebSocket → дашборд.
- **Антимерцание (рекомендуется):** метку `fight` показывать, только если 2-3 окна
  подряд дали `fight` (или EMA по `confidence`). ~5 строк, сильно улучшает UX. Модель не трогаем.

## Живое видео в браузере (ВАЖНО)

RTSP **нельзя** показать в браузере напрямую. Используйте готовое:
**go2rtc** или **MediaMTX** (конвертируют RTSP → WebRTC/HLS). Не изобретайте свой плеер.

## Рекомендуемый стек

- Frontend: **React + TypeScript**.
- Backend + Broker: **Python (FastAPI)** — один язык, переиспользуем OpenCV/RTSP.
- БД: **SQLite** (MVP).

## Чего НЕ делаем в MVP (не усложнять)

- Redis / очереди сообщений, Postgres, аутентификация/роли, мультитенантность,
  тяжёлая аналитика, запись всех клипов. Всё это — после рабочей вертикали.

## Точность (по п.5)

Модель будет «дёргаться» (fight/normal туда-сюда) — для MVP это нормально.
Минимальное сглаживание в брокере (см. выше) убирает большую часть мерцания.
Глубокую стабилизацию — позже.

### События (Event Log) — драки

**Что логируем.** Только подтверждённую драку, не каждый вызов `/detect/fight`.
Один эпизод драки = одна запись в БД (а не одна запись на каждое окно из 16
кадров пока драка продолжается).

**Антимерцание — как именно подтверждаем.**
Модель отвечает на каждое окно независимо, поэтому метка может "дёргаться":
`normal → fight → normal → fight` даже в течение одной реальной драки.
Брокер не пишет событие по первому `fight`, а ждёт подтверждения:

- держит скользящий буфер последних 3 результатов по камере;
- событие создаётся только если **2 из 3 последних окон** дали `fight`
  (или: confidence усреднён по EMA и держится выше порога несколько
  окон подряд — реализация на выбор брокера, суть одна).

**Длительность одного эпизода.** Пока подряд идут подтверждённые `fight` —
это один и тот же эпизод, событие не дублируется. Новое событие создаётся
только после паузы: несколько окон подряд вернули `normal`, и драка
зафиксирована заново (значит — новый инцидент).

**Связь с камерой и пользователем.**

## Порядок работ (рекомендация)

1. Backend: CRUD камер + SQLite + WebSocket (заглушка событий).
2. Frontend: страницы 1-3 против этого backend.
3. Broker: одна камера (можно видеофайл вместо RTSP) → `/detect/fight` → событие → WebSocket → дашборд.
4. Меню модулей (п.4) + оверлей результата (п.5).
5. go2rtc/MediaMTX для живого видео.
